the problem

Launching many agents is powerful. And opaque. And dangerous.

Dynamic workflows spin up dozens of agents in parallel. They get a lot done, but you can't see what they touch, and one irreversible action — deleting data, deploying, rotating keys — can take down production.

The usual reflex is to ask the model, in the prompt, to “be careful” or “ask before deleting.” That's a convention: the model can skip it, get it wrong, or be manipulated by prompt injection. It's not control.

the approach

Invert the trust: control where the agent acts, not where it thinks.

Control isn't placed in the agent's brain (the LLM, infinite and unpredictable), but in the tool through which every action with effects passes. If the gate lives inside the tool, the effect is physically unreachable without crossing it — no matter which model invoked it.

Don't try to certify every driver. Put a checkpoint at the tollgate of the only road that leads to dangerous operations. Whoever wants to act crosses the tollgate.

the four pillars

What it's made of

Agent Kanban board

Each agent is a card that moves across columns. You see live what it's doing, its cost and time, and the results. Multi-dimensional WIP measured in cost: the Working limit is number of agents, total €, and € on foundational models — a local token is ~free, an Opus/GPT-4o one is not.

Two-key approval

Operations with effects stop and wait for your Approve/Deny. Approving requires an operator token: an agent can't self-approve.

Deterministic policy gate

The decision is made by rules in code (policy.json): allow / ask / deny per tool and domain. Reproducible and audited — not negotiable by the model.

Multi-model roles

A different LLM per role: explorer, auditor, implementer, verifier, documenter. A verifier from another family kills the self-approval bias.

LLM-agnostic

Control lives in the tool, so it doesn't matter who calls it

Because the gate is in the tool (via MCP) and not in the client, the same control works for any agent — today's and whatever ships tomorrow. Each one plugs in through its own surface:

  • One MCP server = one domain of functionality (notify, data, deploy…), with least-privilege credentials.
  • The mcp__server__tool convention lets a single policy reason per domain with globs.
  • It governs even the third-party MCPs you can't modify.

honest governance

Powerful, and aware of its limits

  • Deterministic: same tool + same arguments → always the same decision. Made by the code, not the model.
  • Auditable: a hash-chained trail, verifiable (audit.py verify); every decision, whether by rule or human, tied to the payload's hash.
  • With resource and cost limits: aggregate windowed quotas (they close the slicing loophole) and € budgets that distinguish local models (~free) from foundational ones (expensive), both on the board and in quotas (€/hour).
  • Fail-closed: without approval, nothing runs. The system doesn't relax when something fails.
  • With a threat model that enumerates where an action could slip through and how to close it — including what it doesn't yet cover.

The gate governs the actions that pass through the tool, not what the model says or reasons. Documenting the limit is part of closing it well: it avoids a false sense of security.
